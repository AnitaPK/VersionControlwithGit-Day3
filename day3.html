2. Introduction to Git

**Git** is a **Distributed Version Control System (DVCS)** used to track and manage changes to code over time. It allows multiple developers to safely collaborate.

**Analogy:** Think of Git as a **time machine for your code**. It lets you instantly revert to previous states, inspect history, and track who changed what.

Key Concepts

  • **Repository (Repo):** The project database of all files and history.
  • **Commit:** A permanent snapshot of the repository at a specific moment.
  • **Branch:** An independent, parallel line of development.

DVCS Advantage

Every developer has a **full copy** of the project history locally, enabling offline work and reducing dependency on a single central server.

4. Branching: Isolation & Collaboration

**Branching** is the core feature that enables safe, concurrent development. It keeps new features separated from the stable code (`main`).

**Analogy:** Imagine multiple chefs (developers) creating different recipes (features) at separate stations. Once a dish is perfect, they combine it (**merge**) with the main menu.

The Workflow

  1. Create a new branch for a feature (`feature-x`).
  2. Develop and commit changes on the new branch.
  3. Switch back to `main`.
  4. Merge the completed feature branch into `main`.

Protection

The **`main`** branch should always remain stable. If a feature branch breaks, the main project is unaffected.

Use `git branch` to list branches and `git checkout` to switch between them.

Practical 3: Merging With Conflicts (The Reality)

Detecting the Conflict

A **merge conflict** occurs when changes from two branches modify the *same lines* of code.

# 1. Main and feature-2 change the same line of 'config.txt' git checkout main git commit -am "Main change to config" git checkout feature-2 git commit -am "Feature 2 change to config" # 2. Attempt merge into main git merge feature-2 Auto-merging config.txt CONFLICT (content): Merge conflict in config.txt Automatic merge failed; fix conflicts and then commit the result.

Resolving the Conflict

You must manually edit the conflicted file, choosing which code to keep, and then tell Git the conflict is fixed.

# 1. The conflicted file markers: <<<<<< HEAD (main's changes) SERVER_IP=192.168.1.1 ======= (separator) SERVER_IP=10.0.0.5 >>>>>>> feature-2 (feature-2's changes) # 2. Edit 'config.txt' to the correct combined value (e.g., SERVER_IP=10.0.0.5) # 3. Add and commit the resolved file git add config.txt git commit -m "Resolve merge conflict for config update"

Practical 5 & 6: History & Cleanup

Viewing the Commit History

Use `git log` to see the complete history. The `--graph` flag is essential for visualizing branches.

# View compact, graphical history of all branches git log --oneline --graph --all * 123c45d (HEAD -> main) Resolve merge conflict | * 987a6b5 (feature-2) Feature 2 change |/ * c1d2e3f Implement user login logic * 9a8b7c6 Initial commit

Deleting Branches

Once a feature is merged, the branch is obsolete and should be deleted to keep your repository clean.

# Delete a branch that has been successfully merged (safer option) git branch -d feature-1 Deleted branch feature-1 (was c1d2e3f). # Force delete a branch (use only if you're sure you don't need the changes) git branch -D stale-branch
